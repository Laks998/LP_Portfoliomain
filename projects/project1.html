Profile & Booking Flow Redesign

Restructuring the entire booking experience for multi-sport coaches

The Problem

The original design had a fundamental flaw: it placed a single "Book" button at the bottom of the profile. For multi-sport coaches, this forced users through an extra step—after clicking Book, they had to select which sport they wanted before proceeding.

The original wireframes assumed all coaches taught only one sport. However, SportsCove doesn't exclusively work with international-level coaches—the platform also includes retired athletes and lower-level coaches who often have expertise in multiple sports. For example, a boxing coach might also teach Muay Thai, or a yoga instructor might offer pilates.

The existing design hadn't accounted for multi-sport coaches at all, despite this being a reality from day one.

The Solution

I completely restructured the profile and booking flow:

Key Design Changes

Before (Left Screen)
  • Generic "Grade II Boxer" label with no context
  • All sports lumped together with no clear differentiation
  • Single "Individual/Group Session" choice at bottom
  • User had to click Book individual or group → Schedule
  • No visibility into what each sport offering included
After (Middle & Right Screens)
  • Courses Offered section: Each sport displayed as its own card with thumbnail, details, and dedicated Book CTA
  • Sport-specific tabs: Clicking a sport updates the profile to show that sport's experience, certifications, and skill level
  • Self-assessment integration: "Based on the Self-Assessment test, you are in Boxing: Class 2"
  • Detailed class breakdown: Expandable accordions showing exactly what users will learn in each class
  • Direct booking: Book CTA on each course card takes users directly to booking for that specific sport

Design Decisions & Reasoning

Course card separation: Each sport needed its own visual identity. This made it immediately clear that the coach offers multiple distinct services, not just variations of one sport.
Individual book CTAs: Moving the Book button from the profile bottom to each course card eliminated an entire step in the flow. Users' intent became explicit—when they click Book on "Boxing", we know exactly what they want.
Contextual information updates: When users switch between sport tabs, showing sport-specific experience and certifications reinforces expertise and builds trust.
Class transparency: The old design hid what classes entailed. By showing detailed class descriptions with learning outcomes, users could make informed decisions before booking.
Self-assessment integration: Instead of making users guess which class level to book, the self-assessment test automatically places them, reducing anxiety and wrong bookings.

Information Architecture

The profile structure now follows this hierarchy:

Courses Offered (primary) About / Background Testimonials Certificates

When a sport tab is selected, these update contextually:

About(Years of Exp, training details etc) Certifications Testimonials

Addressing the Booking Friction

Original Flow

Browse profile → Click Book (bottom) → Select sport from list → Click Book required session type → Schedule

New Flow

Browse courses → Click Book on specific sport card → Choose the required session type tab → Schedule

By surfacing missing requirements and simplifying the flow, we reduced ambiguity and enabled faster, more confident user actions.

Additional Improvements

  • Session type flexibility: Individual and Group sessions are now shown upfront and can be easily switched between, unlike the previous flow where they were treated as separate paths requiring multiple steps to change.
  • Class structure clarity: Individual classes and Group classes are clearly differentiated, with their respective structures presented without forcing users to backtrack.
  • Pricing transparency: Class-specific pricing is displayed clearly before booking, allowing different costs for different classes.
  • Premium upsell: "Become a Premium learner now!" is positioned strategically after users are already engaged with the content.

Impact

Easy switching between Individual and Group sessions
Clear, class-specific pricing displayed before booking
Better class transparency
Scalable for any number of sports

Multi-Sport Coach Cards

Context-aware discovery that adapts to user intent

The Problem

How to represent multi-sport coaches clearly on search and discovery cards without cluttering the interface? This became critical once we redesigned the booking flow to support multiple sports per coach.

Initial Exploration

I explored different approaches to display multi-sport coaches on discovery cards:

Option 1: Show all sports on card

Listing "Boxing, Muay Thai, Kickboxing" created visual clutter and reduced scannability, especially on mobile.

Option 2: Show one sport with (+3) indicator

Showing "Boxing (+3)" was ambiguous—users didn't know what the "+3" meant. More critically, if a user searched for "Muay Thai" and saw a coach card showing "Boxing (+3)", they'd likely skip it, missing a relevant result.

Card Display Solution

I implemented context-aware card displays that adapt based on how the user arrived at the coach:

When searching by sport

Card highlights the searched sport first (e.g., "Muay Thai (+1)"), ensuring relevance to user intent

When browsing directly

Card shows the coach's primary/most skilled sport (e.g., "Boxing (+1)"), reflecting their core expertise

This solved the discovery problem—users always saw relevant information in the right context.

Impact

Cleaner, more scannable cards
Clear search intent alignment

Push Notifications System

Designing for clarity, role-awareness, and minimal fatigue

The Challenge

SportsCove manages time-sensitive activities: classes, cancellations, subscriptions, tasks, and events. Users needed critical updates without being overwhelmed. The complexity increased because a single user could act as both learner and coach, while the product operated under early-stage development constraints.

Goal: Design a push notification system that was clear, role-aware, minimal, and scalable.

Research & Exploration

I mapped every possible notification trigger to understand the full landscape:

Class-related
  • Class booked
  • Upcoming class reminders
  • Class cancelled by coach
  • Class cancelled by learner
  • Class rescheduled
  • Class completed
  • Missed session
Subscription-related
  • Subscription purchased
  • Subscription renewal confirmation
  • Subscription expiring soon
  • Subscription expired
  • Payment failure
  • Auto-renewal success
Coach-related
  • New booking received
  • Booking cancelled
  • Schedule changes
  • Payout initiated
  • Payout completed
  • New post by coach
Platform / System
  • Profile updates
  • New features or announcements
  • Promotional messages
  • A new task added
  • A new event drop
  • New message

This exercise revealed which notifications were critical versus noise-inducing.

Decision Framework

I evaluated each notification using five criteria:

User impact: Would missing this cause loss, confusion, or missed income?
Time sensitivity: Does this require immediate or scheduled action?
Role relevance: Is this for the user as a learner or coach?
Frequency & fatigue: Would this become annoying over time?
Development cost: Does the value justify the engineering effort?

The Solution

Only essential, high-impact notifications were retained:

Upcoming class reminder
Class cancelled by coach
Class cancelled by learner
Class rescheduled
Subscription expiring soon
Subscription expired
Payment failure
New booking received
Booking cancelled
Schedule changes
New message

All other notifications were removed or handled in-app only.

Role-Based Clarity

To prevent confusion for dual-role users, I designed role-specific notification formats:

Role-Based Greeting Format
  • Notifications began with addressing the user as"Coach" or "Learner"
  • Subscription notifications omitted role prefixes (universally applicable)
  • Clear wording ensured immediate context recognition

User Control

Optional notifications

Class reminders could be toggled on/off from settings

Critical notifications

Cancellations, subscription expiry, payment failures, new bookings, and schedule changes remained always-on

Timing Strategy

Upcoming classes: 24 hours and 2 hours before
Subscription expiry: 3 days and 1 day before
Cancellations/reschedules/schedule changes: Immediate
New bookings: Immediate
New messages: Immediate (batched if multiple)

No repeated notifications unless status changed.

Impact

Reduced notification fatigue
Clear role context
Reliable critical information
Low maintenance overhead

Core Solutions Delivered

Streamlined User Flows

Redesigning learner and coach experiences end-to-end

The original wireframes scattered critical tasks across disconnected screens with no clear user journeys. I redesigned the entire experience for both learners and coaches across every touchpoint—from onboarding to daily interactions—focusing on reducing cognitive load and creating intuitive pathways.

Onboarding: Designed progressive disclosure flows that gathered essential information without overwhelming new users, with role-specific paths for learners vs coaches
Discovery & Booking: Consolidated search, filtering, and booking into cohesive journeys with clear progress indicators and contextual help
Messages: Streamlined communication between learners and coaches with threaded conversations, quick actions, and booking-related context
Tasks & Events: Created unified task management and event discovery systems that kept users informed without notification fatigue
Subscriptions: Designed transparent subscription flows with clear value propositions, flexible management, and no hidden steps
Social Features: Built community engagement flows (posts, interactions, profiles) that fostered connections while maintaining focus on the core learning experience
Coach-specific flows: Simplified schedule management, earnings tracking, and learner communication into unified workflows that reduced administrative overhead
Key principle: Every flow was designed to minimize steps while maintaining necessary context—removing friction without sacrificing clarity or control

Guest Mode

Explore before committing

Expected impact: 40% increase in sign-ups, reduced bounce rate

Design Process

Iterative Refinement

Weekly sprints with developers, testing prototypes and incorporating feedback. This reduced development rework by 30%.

Key Learnings

Strategic thinking over execution: The most impactful work happened in decision-making frameworks, not pixel-pushing. Understanding constraints and trade-offs shaped better outcomes.

Systems thinking pays dividends: Building the design system upfront saved exponentially more time than it took to create.

Test assumptions early: Usability testing caught critical issues that would have been expensive to fix post-development.
